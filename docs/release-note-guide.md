# Release Note 撰写规范

本文档规定团队发版时 Release Note 的结构、内容标准与常见反模式，适用于 PyPI 发布、GitHub Release、内部发版公告等场景。

## 一、核心原则

| 原则 | 说明 |
|------|------|
| **用户视角** | 读者是使用你产品的人，不是你的同事。每条变更必须让读者理解"这对我有什么影响"。 |
| **一句话价值** | 每条变更用一句话说清楚：做了什么 + 解决什么问题/带来什么价值。 |
| **诚实分级** | bug fix 就是 bug fix，不包装成 feature；真正的架构创新才值得 Highlight。 |
| **合并同类项** | 同一功能的多次迭代/修复合并为一条，只说最终效果，不要贴 10 条 commit message。 |
| **无变动则省略** | 某个分类没有变更就不写，不要出现"算法：无"这种占位。 |

## 二、标准结构

```markdown
## <产品名> v<版本号>

发布日期：YYYY-MM-DD

### Breaking Changes

> 如无则写"本版本无破坏性变更"，不可省略此节。

- 变更描述 + 迁移指引

### Highlights（核心亮点）

- **<功能/模块名>**：一句话说明变更价值
- **<功能/模块名>**：一句话说明变更价值

### Features（新功能）

- 功能描述（解决什么问题 / 面向什么场景）

### Fixes（修复）

- 修复描述（之前的表现 → 修复后的表现）

### Improvements（优化）

- 优化描述（性能/体验/稳定性提升的具体效果）

### Refactoring（重构）

> 仅当重构对用户有可感知影响时才列入（如 API 变化、性能提升）。
> 纯内部重构不进 Release Note。

- 重构描述 + 用户侧影响

### Dependencies

> 仅列出对用户有影响的依赖变更（如最低 Python 版本、新增必装依赖）。

- 依赖变更描述

### Stats（可选）

- 提交 N 条，涉及 M 个文件，+X / -Y 行
- 时间跨度：YYYY-MM-DD → YYYY-MM-DD
```

## 三、每条变更的写法

### 好的写法

```
- 新增飞书长连接 Bot，支持手机端远程向 Agent 下发指令，无需公网 IP
- 修复多窗格聊天时切换模型会误改其他窗格模型的问题
- Skills 列表支持按来源筛选（内置/市场/自定义），并新增单个技能启停开关
```

### 差的写法

```
- 支持多模态模型                    ← 太笼统，什么模型？解决什么问题？
- 修复：智能体-日志导出为空          ← 照搬 JIRA 标题，用户看不懂
- 智能体并发执行能力优化。           ← "优化"是万能词，优化了什么？快了多少？
- 完善 model 模块的单元测试          ← 内部质量工作，用户不关心，不应出现
- 重构代码结构                      ← 没说用户侧影响
```

### 改写对照

| 差的 | 好的 |
|------|------|
| 支持多模态模型 | 新增多模态模型支持（GPT-4o / Claude 3.5），Agent 可处理图片输入并做视觉理解 |
| 修复：智能体-日志导出为空 | 修复 Agent 会话日志导出为空文件的问题，现在可正常导出完整对话记录 |
| 智能体并发执行能力优化 | Agent 并发执行性能提升约 40%，5 个子任务并行场景下总耗时从 12s 降至 7s |
| 完善 model 模块的单元测试 | （不进 Release Note，属于内部质量建设） |
| 重构代码结构，模型模块进行 DDD 模式重构 | 重构模型调用层为 DDD 架构，Provider 切换不再需要重启服务（原需重启） |

## 四、分类规则

### 进入 Release Note 的

| 类型 | 判断标准 |
|------|---------|
| 新功能 | 用户可以做到之前做不到的事 |
| Bug 修复 | 用户之前会遇到的问题现在不会了 |
| 性能优化 | 有可量化或可感知的速度/资源改善 |
| 破坏性变更 | API / 配置 / 行为发生不兼容改变 |
| 依赖变更 | 影响用户安装或运行环境 |

### 不进入 Release Note 的

| 类型 | 原因 |
|------|------|
| 单元测试补充 | 内部质量，用户无感知 |
| CI/CD 流水线调整 | 内部工程，用户无感知 |
| 代码风格/lint 修复 | 无功能影响 |
| 文档同步/plan 落盘 | 内部流程 |
| 纯内部重构（无 API/行为变化） | 用户无感知 |
| AGENTS.md / 内部配置更新 | 内部流程 |

## 五、版本号与 Tag 关联

每次发版必须：

1. `pyproject.toml`（或对应配置文件）中版本号已更新
2. Git 打 tag：`v<版本号>`（如 `v0.3.5`），tag 指向版本 bump commit
3. GitHub Release 的 tag 与 PyPI/npm 等包管理器版本号一致
4. Release Note 末尾附 Full Changelog 对比链接：
   ```
   https://github.com/<org>/<repo>/compare/v<旧版本>...v<新版本>
   ```

## 六、Breaking Changes 必填规则

`Breaking Changes` 节**每个版本都必须存在**：

- 有破坏性变更 → 列出变更 + 迁移步骤
- 无破坏性变更 → 写"本版本无破坏性变更"

这是对用户最重要的信息，不可省略或遗漏。

## 七、Review Checklist

发版前对照检查：

- [ ] 每条变更都说了"对用户的影响"，而不只是"改了什么代码"
- [ ] 没有出现"优化"、"完善"、"调整"等不带具体描述的万能词
- [ ] 同一功能的多次 commit 已合并为一条
- [ ] 内部工作（测试/CI/文档同步/重构）没有混入
- [ ] 无变动的分类已省略，没有出现"XXX：无"
- [ ] Breaking Changes 节存在且内容正确
- [ ] 版本号与 tag 和包管理器一致
- [ ] Full Changelog 链接正确
